# DID 设计建议

## 1. 目标

DID 设计的核心不是“把所有变量都暴露出来”，而是建立一套：

- 稳定
- 可演进
- 能区分访问权限
- 能兼顾诊断和产线需求

的标识体系。

## 2. 建议分层

推荐将 DID 分成以下几类：

| 类型 | 用途 | 建议 |
|---|---|---|
| 固定身份类 | SN、版本、硬件型号 | 只读，跨版本尽量稳定 |
| 运行状态类 | 温度、电压、状态字 | 只读，适合 0x22 / 0x2A |
| 配置参数类 | 阈值、使能开关、标定参数 | 读写分权，建议配合 0x27 |
| 调试/工厂类 | 临时调试参数、内部开关 | 单独编号段，量产可裁剪 |
| 虚拟控制类 | 与 0x2F 配套 | 避免与普通参数 DID 混用 |

## 3. 编号建议

可以按号段规划，例如：

```text
0x0100 ~ 0x01FF   板级 I/O / demo DID
0x1000 ~ 0x1FFF   产品基础信息
0x2000 ~ 0x2FFF   实时状态量
0x3000 ~ 0x3FFF   可写配置参数
0x4000 ~ 0x4FFF   工厂/调试参数
0xF000 ~ 0xF0FF   Routine / Console 相关保留区
```

## 4. 与本仓库代码的映射建议

### 4.1 0x22 / 0x2E

`service_0x22_0x2E_param.c` 更适合承载“配置类 DID / 状态类 DID”。

### 4.2 0x2F

`service_0x2F_io.c` 更适合承载“控制类 DID”，例如：

- LED
- 继电器
- PWM 输出
- 模拟量强制输出

不要把“普通配置参数写入”和“实时控制”混在同一个 DID 上。

## 5. 数据格式建议

- 明确大小端
- 明确单位与缩放倍率
- 明确只读/可写属性
- 明确是否要求扩展会话 / 安全解锁
- 为每个 DID 维护独立文档或表格

## 6. 推荐最小文档字段

| 字段 | 说明 |
|---|---|
| DID | 16bit 编号 |
| 名称 | 业务名称 |
| 方向 | R / W / RW / IO |
| 长度 | 固定或变长 |
| 编码 | 整数/ASCII/bitfield/结构体 |
| 单位 | V、mA、℃ 等 |
| Session 要求 | default / extended / programming |
| Security 要求 | 是否需解锁 |
| 备注 | 兼容性或历史说明 |
